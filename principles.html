Design Principles

Visual, interaction, and UX principles derived from the Prometheus codebase.

On this page
Visual Principles Interaction Principles UX Heuristics Product Tone Composition & Escape Hatches

Visual Principles

Data-first, minimal chrome

Every pixel of interface should serve the user's monitoring task. The Prometheus UI is a developer tool — decoration is noise. Borders, shadows, and colors exist only to delineate data and convey health state.

System fonts by design

The design system intentionally uses the system UI font stack. This is not laziness — it's a deliberate choice for a developer tool that should feel native to each platform. No font loading means faster renders for dashboards with thousands of elements.

-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, Helvetica, Arial, sans-serif

Monospace for metrics

DejaVu Sans Mono is the Prometheus-specific monospace choice. It's used for PromQL expressions, metric names, label values, configuration YAML, and numeric data displays.

Health colors are semantic

The five-state health color system (ok/err/warn/info/unknown) maps to green/red/yellow/blue/gray. These are never used decoratively — they always convey system health or operational state.

The header is always dark

The rgb(65, 73, 81) header is intentionally mode-independent. It's part of Prometheus brand identity and anchors the interface. The header never adapts to light/dark mode switching.

Interaction Principles

URL-persistent state

All major filter states — search text, health/state filters, pagination, active tab — are persisted in the URL via query parameters. This enables:

  • Bookmarkable views of filtered targets or alerts
  • Shareable links that reproduce the exact dashboard state
  • Browser back/forward navigation through filter history

Fuzzy search everywhere

All filter inputs use fuzzy matching (via @nexucis/kvsearch). Users don't need to remember exact metric or label names — partial matches surface results naturally.

Progressive disclosure

Information is layered: accordion panels show summary rows that expand into full detail. Target labels show active labels by default with a toggle to reveal pre-relabeling discovered labels. The PromQL tree view executes queries node-by-node, bottom-up.

Collapse all / expand all

Pages with many accordion panels (Targets, Service Discovery) provide a single button to collapse or expand all panels. This is critical when monitoring hundreds of scrape targets.

UX Heuristics

Performance-conscious rendering

  • LabelBadge uses plain <span> elements instead of Mantine Badge for high-throughput rendering (thousands of labels)
  • SeriesName falls back to plain text mode when series count exceeds 1,000
  • CustomInfiniteScroll limits rendered DOM nodes in long lists
  • Heavy filtering uses useMemo on rendered lists

Suspense-driven loading

All async data uses useSuspenseAPIQuery wrapped in coordinated Suspense + ErrorBoundary pairs. This ensures:

  • Consistent loading skeletons across all pages
  • Error boundaries that reset on route change
  • No manual loading state management

Old data stays visible

The graph component keeps previous data visible while a new query is in-flight. No flash-to-empty between time range changes.

Product Tone

Technical and precise

Labels say "err" not "Error occurred". Badges display the raw status string ("firing", "up", "down") without translation. The interface trusts that its users are infrastructure engineers who understand these terms.

Infrastructure-focused

The vocabulary is monitoring-native: "scrape targets", "rule groups", "label matchers", "PromQL expressions". The design system preserves this domain language rather than abstracting it away.

No marketing polish

There are no hero sections, no gradient backgrounds, no decorative illustrations. The Prometheus UI is a tool, not a product landing page. The design system enforces this restraint.

Composition & Escape Hatches

A design system is only useful if you can work around it when it doesn't fit. The Prometheus Design System is designed with explicit escape hatches at every level.

className and style props

All components accept className and style props. You can always override or extend any component's appearance at the CSS level without forking the component.

// Override StatusBadge styling for a specific context
<StatusBadge
  status="up"
  className="my-custom-badge"
  style={{ fontSize: '0.7rem' }}
/>

CSS variables work standalone

The --prom-* CSS custom properties can be consumed independently of the React components. If your page or component doesn't use React, you can still adopt the design tokens for consistent colors, spacing, and typography — just include the token stylesheet.

/* Use tokens without importing any React component */
.my-element {
  color: var(--prom-health-ok-text);
  border-left: 3px solid var(--prom-health-ok-border);
  padding: var(--prom-spacing-md);
  font-family: var(--prom-font-mono);
}

What the design system doesn't cover

The design system intentionally excludes the PromQL editor and related CodeMirror-based components. These remain in application code because they are tightly coupled to the query engine and have specialized performance requirements (syntax highlighting, autocompletion, tree visualization) that don't generalize into a design system.

When a component doesn't fit

If a design system component doesn't meet your requirements after trying className/style overrides:

  1. Copy-paste the source — every component is a single file with no internal dependencies beyond Mantine and the token system. Fork it into your app code rather than adding workaround props.
  2. File an issue — if the limitation is likely to affect others, open an issue describing the use case. It may lead to a better component API.
  3. Propose a variant — follow the contributing process to propose a new variant or component.
