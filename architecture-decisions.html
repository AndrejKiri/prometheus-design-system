Architecture Decisions

Technical rationale behind the design system's key choices.

On this page
Why Mantine? Bundle Size Impact Token Independence Upgrade Policy No Custom Fonts

Why Mantine?

Mantine 8 was chosen as the component foundation for one reason: it's already in the Prometheus UI. The Prometheus 3.x web interface is built on Mantine, making it the derived — not invented — choice.

Alternatives considered

LibraryWhy not
Radix + custom CSSWould require rebuilding every component from scratch. The existing codebase already uses Mantine's component APIs — switching would be a rewrite, not an extraction.
MUI (Material UI)Different design language (Material Design). Would impose a visual identity that doesn't match Prometheus's information-dense, developer-tool aesthetic.
Chakra UISimilar story to MUI — opinionated visual defaults that would fight the existing Prometheus look and feel.
No library (vanilla)The existing codebase already depends on Mantine. Removing it would be an invasive refactor with no clear benefit for maintainers.

What Mantine provides

  • Accessible primitives (Accordion, Select, TextInput) with keyboard navigation
  • Built-in color scheme support (light-dark() CSS function)
  • CSS Modules integration — no runtime CSS-in-JS
  • Responsive props system (hiddenFrom, visibleFrom)
  • TypeScript-first API with full type coverage

Bundle Size Impact

The design system adds minimal overhead beyond what the Prometheus UI already bundles:

PackageGzip SizeNotes
@prometheus-io/design-system~12 KB19 components + token definitions
@mantine/core~45 KBAlready a peer dep — no new cost
@tabler/icons-reactTree-shakeableOnly used icons are bundled
CSS tokens (--prom-*)~2 KBStandalone, no JS required

Performance-conscious components

Several components deliberately avoid Mantine wrappers where performance matters:

  • LabelBadge uses plain <span> elements instead of Mantine Badge — critical when rendering thousands of labels on the Targets page
  • SeriesName falls back to plain text mode above 1,000 series to avoid DOM explosion
  • All list rendering uses useMemo and virtualization patterns where appropriate

Token Independence

The --prom-* CSS custom properties are designed to be consumed without the React components:

<!-- Use tokens without React -->
<link rel="stylesheet" href="@prometheus-io/design-system/tokens.css">

<!-- Then in your CSS -->
.my-status {
  background: var(--prom-health-ok-bg);
  color: var(--prom-health-ok-text);
  font-family: var(--prom-font-body);
}

This means any part of the Prometheus UI — including non-React pages, documentation, or third-party integrations — can adopt visual consistency without a React dependency.

Upgrade Policy

Mantine major updates

When Mantine releases a new major version:

  1. The design system will be tested against the new version within 2 weeks of release
  2. A compatibility report will be published as a GitHub issue
  3. Breaking changes will be absorbed by the design system — downstream consumers should not need to change component usage
  4. If breaking changes are unavoidable, they will be documented in the Changelog with migration instructions

Semantic versioning

  • Patch (0.1.x) — bug fixes, dark mode improvements, no API changes
  • Minor (0.x.0) — new components, new token values, backward-compatible additions
  • Major (x.0.0) — breaking prop changes, removed components, Mantine major upgrade passthrough

No Custom Fonts

The decision to use the system UI font stack is deliberate:

  • Zero font loading time — critical for a monitoring dashboard that may render thousands of elements
  • Native platform feel — Prometheus is a developer tool; it should feel like part of the OS, not a branded product
  • No FOUT/FOIT — no flash of unstyled or invisible text during page load
  • Smaller bundle — no font files to serve or cache

DejaVu Sans Mono is the sole exception — it's the Prometheus-specific monospace choice used for PromQL expressions, metric names, and code blocks.
