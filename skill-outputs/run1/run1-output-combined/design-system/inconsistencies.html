Inconsistencies

9 inconsistencies identified during Phase 1 visual audit — INC-001 through INC-009

Each entry includes the observed variation, canonical resolution, and a link to the corresponding action item.
INC-001 · cosmetic · style-drift
Card titles inconsistently include a leading icon
Every card title should either consistently include a leading icon (matched to the card's semantic) or consistently omit one. Mixing both within the same app creates an uneven visual rhythm for what is semantically the same container pattern.
/status and /alertmanager-discovery place a Tabler icon immediately before the card title, while /tsdb-status and /flags render the title as plain text with no icon. The underlying Mantine Card wrapper is identical, so the variation is a header-composition choice rather than a different component. Users scanning the app see the icon on some cards and expect it elsewhere — its absence reads as missing information.
  • /status — Card title 'Build information' has a leading DB icon; 'Runtime information' has a leading running-figure icon.
  • /alertmanager-discovery — Card title 'Active Alertmanagers' has a leading bell icon; 'Dropped Alertmanagers' has a leading crossed-bell icon.
  • /tsdb-status — Card titles 'TSDB Head Status' and 'Top 10 label names with value count' render as plain text with no leading icon.
  • /flags — The page-level card title renders as plain text with no leading icon.

Pages: /status, /tsdb-status, /flags, /alertmanager-discovery

INC-002 · minor · style-drift
Code/identifier text uses monospace on some pages but sans-serif on others
Label names, metric names, and flag names are code-like identifiers and should consistently use the monospace font family so they are visually distinguishable from prose.
/flags wraps each flag name in a <code> element which inherits a monospace family, while /tsdb-status renders equivalent identifiers (__name__, label=value pairs) as plain <td> text in the body sans-serif stack. The content is the same semantic class — a literal string that is not prose — but it is rendered with different font families on different pages.
  • /flags — Flag names are wrapped in <code> tags; font-family resolves to browser default 'monospace'.
  • /tsdb-status — Identifiers like __name__ and label=value pairs are plain <td> text; font-family is the system sans-serif stack (-apple-system, ...). Visually looks monospace-ish because of the characters used but is actually proportional.

Pages: /flags, /tsdb-status

INC-003 · cosmetic · style-drift
Label badges preserve case but stats badges uppercase — same visual, divergent rules
Two badge classes that share the same background, padding, and radius should share the same text-casing rule, or be visually differentiated so users can learn which is which.
On /targets, the _labelBadge class (used for scrape-target labels) has textTransform: none, while _statsBadge (used for scrape-pool statistics) has textTransform: uppercase. Both badges share the same grey background rgb(52,58,64) in dark / equivalent light grey, same radius (~1000px), same padding. The only visual differentiator is the casing — which is triggered by CSS rather than authored content, making it brittle.
  • /targets — _labelBadge: textTransform: none. Content like 'instance=localhost:9090' preserves case as authored.
  • /targets — _statsBadge: textTransform: uppercase. Content authored as lower/mixed case ('endpoints', 'healthy') is displayed uppercased ('ENDPOINTS', 'HEALTHY').

Pages: /targets

INC-004 · minor · style-drift
Filter inputs split between multi-pill toolbar and single-field text input
All 'filter this list' surfaces should use the same input pattern so users transfer their mental model between pages. Pick either the multi-pill toolbar (for label-oriented filtering) or the single text field (for substring filtering) and apply it everywhere.
/alerts, /rules, /targets, and /service-discovery render their filter as a Mantine PillsInput multi-filter toolbar — users add chip-style labeled filters. /flags instead uses a plain full-width Mantine TextInput with a leading search icon. The surfaces are conceptually identical (narrow a list to matching rows) but the interaction is different. This is notable because /flags's data (flag name + value) would benefit from the same labeled-filter approach used on /targets.
  • /alerts — PillsInput multi-filter toolbar with placeholder 'Filter by rule name or labels'.
  • /rules — PillsInput multi-filter toolbar with placeholder 'Filter by rule name or labels' (identical to /alerts).
  • /targets — PillsInput multi-filter toolbar scoped to target labels.
  • /service-discovery — PillsInput multi-filter toolbar scoped to discovered labels.
  • /flags — Single full-width Mantine TextInput with a leading search icon, placeholder 'Filter by flag name or value'.

Pages: /alerts, /rules, /targets, /service-discovery, /flags

INC-005 · minor · implementation-drift
Only one table is sortable; the other tables are static
Tables that display list data should offer the same interactive affordances (sort by header) consistently, so users don't have to guess where sorting is available.
/flags's Flag/Value table has sortable column headers with a stacked chevron-up + chevron-down icon (Mantine sort control). /status and /tsdb-status render similar two-column key/value tables but the headers are non-interactive. The user cannot tell at a glance which tables support sorting without clicking.
  • /flags — Table headers show stacked chevron-up + chevron-down icons (class _control_1w9xk_5) and reorder rows on click.
  • /status — Key/Value table headers are plain text, no sort icon, no click-to-sort behavior.
  • /tsdb-status — Name/Count table headers are plain text, no sort icon, no click-to-sort behavior.

Pages: /flags, /status, /tsdb-status

INC-006 · minor · missing-normalization
Success states rendered inconsistently as plain text vs colored badges
Success/warning/error states should consistently use the same visual treatment (a colored status pill) so users can scan status information across pages at a glance.
On /status, the line 'Configuration Reload: Successful' displays the status as plain prose in body color. Elsewhere in the app (/targets Health Status Badge), equivalent semantic states are shown as green/red pills with rounded pill styling. This is the same semantic information (an 'OK' signal) rendered with two different visual weights.
  • /status — 'Configuration Reload: Successful' rendered as plain <td> text in the body color. No pill, no color differentiation.
  • /targets — Target health 'up' rendered as a green pill with _healthOk class — dedicated background color, uppercased, pill radius.

Pages: /status, /targets

INC-007 · minor · accessibility
Health badge source text ('up'/'down') does not match rendered text ('UP'/'DOWN')
Content should be authored in the form the user sees. Relying on CSS textTransform to uppercase lowercase source text makes copy-paste, screen-reader output, and in-page search inconsistent with what is visible on screen.
The DOM contains the literal lowercase strings 'up' and 'down' inside the _healthOk / _healthErr spans on /targets, but CSS textTransform: uppercase (inherited via the health-badge class) renders them as 'UP' and 'DOWN'. A user searching the page for 'UP' via browser find-in-page will get zero matches; a screen reader may announce the lowercase form. This is a minor accessibility and interop concern.
  • /targets — Source DOM: <span class='_healthOk'>up</span>. Rendered on screen: 'UP' (textTransform: uppercase).
  • /targets — Source DOM: <span class='_healthErr'>down</span>. Rendered on screen: 'DOWN' (textTransform: uppercase).

Pages: /targets

INC-008 · cosmetic · style-drift
highlight.js token colors swap roles between light and dark themes
The same syntactic category (a YAML key, a YAML string value) should be assigned to the same hue family across themes — only lightness/saturation should shift. Swapping 'keys are green' in light to 'keys are blue' in dark (and vice versa for strings) breaks token memory across theme switches.
In light mode on /config, hljs-attr (YAML keys) is olive-green rgb(92,148,13) and hljs-string is blue rgb(24,100,171). In dark mode, hljs-attr becomes light-blue rgb(165,216,255) and hljs-string becomes green rgb(64,192,87). The hues are swapped between the two semantic categories when the theme changes. Both directions are readable, but a user who has learned 'green = key' in light mode will see 'green = string' in dark mode.
  • /config — Light theme: hljs-attr rgb(92,148,13) olive-green; hljs-string rgb(24,100,171) blue.
  • /config — Dark theme: hljs-attr rgb(165,216,255) light-blue; hljs-string rgb(64,192,87) green.

Pages: /config

INC-009 · minor · style-drift
<pre> code block relies on browser default 'monospace' rather than an explicit font stack
Code blocks should declare an explicit monospace font stack (e.g., 'ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, monospace') so the rendered glyph width, weight, and shape are consistent across macOS, Windows, and Linux.
On /config, the <pre> element's computed font-family is exactly the single keyword 'monospace', which is whatever the browser treats as its default. On macOS that resolves to one font, on Windows another, on Linux yet another — producing visibly different renderings of the same config across users. Adding an explicit stack is a one-line fix.
  • /config — Computed style on <pre>: font-family: monospace (single keyword — browser default).
  • /flags — Flag names wrapped in <code> — also inherits the browser default 'monospace' with no explicit stack. Same issue, different surface.

Pages: /config
