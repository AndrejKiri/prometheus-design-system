Contributing to the Design System

How to propose, build, and submit changes to the Prometheus Design System.

On this page
Proposing a New Component How Design Decisions Are Made Development Setup PR Review Process Code Standards

Proposing a New Component

Before building a new component, open a Component Proposal issue in the GitHub repository with the following information:

FieldDescription
Component nameWhat you'd call it (e.g., HistogramChart)
Existing usageWhere this pattern already exists in prometheus/prometheus (link to source files)
Problem it solvesWhat inconsistency, duplication, or gap does it address?
Proposed APITypeScript interface showing the props you'd expect
VariantsHow many visual or behavioral variants are needed?
Performance notesAny rendering constraints (e.g., "must handle 10k items")
Derive, not invent
Components must be derived from existing patterns in the Prometheus codebase. Proposals for components that don't exist anywhere in the current UI will be asked to first submit the pattern as an upstream PR to prometheus/prometheus.

How Design Decisions Are Made

  1. Proposal — open an issue describing the change and its rationale
  2. Discussion — maintainers and community members comment, ask questions, suggest alternatives
  3. Consensus — following Prometheus project norms, decisions are made by consensus, not voting. If maintainers agree the change is consistent with the "derive, not invent" philosophy, it proceeds.
  4. Implementation — a PR is opened following the code standards below
  5. Review — at least one maintainer reviews the code, docs, and visual output
  6. Documentation — the component page, changelog, and any affected architecture decisions are updated as part of the PR

Development Setup

# Clone the repository
git clone https://github.com/andrejkiri/prometheus-design-system.git
cd prometheus-design-system

# Install dependencies
npm install

# Start the development server (Storybook)
npm run storybook

# Run the test suite
npm test

# Build the documentation site
npm run docs:build

# Lint and type-check
npm run lint
npm run typecheck

PR Review Process

What a component PR should include

  1. Component implementation (.tsx + .module.css)
  2. Storybook story showing all variants and states
  3. Unit tests for logic and accessibility
  4. Documentation page update
  5. Changelog entry

Review checklist

CheckRequirement
DerivationComponent is derived from existing Prometheus UI code — link to source required
Dark modeWorks correctly in both light and dark mode using light-dark()
PerformanceTested with realistic data volumes (100+ items where applicable)
AccessibilityKeyboard navigable, screen reader friendly, WCAG AA contrast
TypeScriptFull type coverage, exported interface for props
TokensUses --prom-* tokens — no hardcoded colors or spacing values
DocumentationProps table, usage examples, variant demos on the docs site

Code Standards

Component structure

  • One component per directory under src/components/
  • CSS Modules for styling (.module.css) — no inline styles except for dynamic values
  • All components accept className and style as escape hatches
  • Export a TypeScript interface for props (not just type)

Naming conventions

  • PascalCase for components: StatusBadge, HealthPanel
  • camelCase for props: showCollapseAll, defaultStatus
  • --prom- prefix for all CSS custom properties
  • Descriptive, specific names — FilterToolbar not Toolbar

CSS conventions

  • Use light-dark() for all mode-dependent colors
  • Reference --prom-* tokens instead of Mantine color steps directly
  • Avoid !important — if overrides are needed, the component API is insufficient
  • Use Mantine's spacing scale (xs, sm, md, lg, xl) — don't invent new sizes
